# 需求文档

## 简介

校园墙是一个面向大学生的社区互动平台，提供表白墙、树洞、求助、市集、失物招领等功能模块。系统采用 Vue 3 + DaisyUI 构建前端，Java 21 + Sa-Token 构建后端，使用 PostgreSQL 存储数据，Redis 进行缓存和会话管理。

## 术语表

- **Campus_Wall_System**: 校园墙系统，本项目的核心应用
- **User**: 已注册并通过身份验证的校园用户
- **Post**: 用户发布的内容，包含文字和可选图片
- **Board**: 内容分区板块（表白墙、树洞、求助、市集、失物）
- **Moderator**: 版主，拥有内容审核权限的用户
- **console**: 管理员，拥有系统管理权限的用户
- **Sa-Token**: Java 轻量级权限认证框架
- **DaisyUI**: 基于 Tailwind CSS 的 UI 组件库

## 功能性需求

### 需求 1: 用户认证

**用户故事:** 作为 校园用户，我想要 注册和登录系统，以便于 我可以发布和互动内容。

#### 验收标准

1. 当 用户提交有效的注册信息 时，系统应 创建新账户并返回认证令牌
2. 当 用户提交有效的登录凭证 时，系统应 通过 Sa-Token 验证用户身份并建立会话
3. 当 用户提交无效的凭证 时，系统应 拒绝请求并返回错误信息
4. 当 用户请求登出 时，系统应 使当前会话失效并清除认证状态
5. 当 用户会话过期 时，系统应 要求重新认证后才能执行受保护的操作
6. 当 用户使用 EDU 邮箱完成验证 时，系统应 标记该用户为"已认证校友"并在个人资料中显示认证标识
7. 当 未验证身份的用户尝试发帖 时，系统应 提示用户先完成校园身份验证
8. 当 用户无法使用 EDU 邮箱验证 时，系统应 提供学生证/录取通知书人工审核通道作为替代方案
9. 当 同一学号尝试注册多个账号 时，系统应 拒绝注册并提示该学号已被使用
10. 当 新生用户提交录取通知书验证 时，系统应 授予"准验证"状态，允许浏览和在"新生咨询"板块发帖

### 需求 2: 首页展示

**用户故事:** 作为 用户，我想要 在首页查看各板块的热门内容，以便于 我可以快速了解校园动态。

#### 验收标准

1. 当 用户访问首页 时，系统应 展示来自所有板块的热门帖子聚合
2. 当 用户查看首页 时，系统应 显示通往所有板块的快捷导航链接
3. 当 有新帖子发布 时，系统应 在30秒内更新首页内容
4. 当 用户点击帖子预览 时，系统应 导航到完整的帖子详情页

### 需求 3: 表白墙板块

**用户故事:** 作为 用户，我想要 在表白墙发布和浏览表白内容，以便于 我可以表达情感或寻找有缘人。

#### 验收标准

1. 当 用户创建表白帖子 时，系统应 根据用户偏好以匿名或实名方式保存帖子
2. 当 用户浏览表白墙 时，系统应 按创建时间降序显示帖子
3. 当 用户与表白帖子互动 时，系统应 允许点赞和评论操作
4. 当 用户设置匿名模式 时，系统应 对其他用户隐藏作者身份

### 需求 4: 树洞板块

**用户故事:** 作为 用户，我想要 在树洞匿名分享心事，以便于 我可以倾诉而不暴露身份。

#### 验收标准

1. 当 用户创建树洞帖子 时，系统应 默认强制匿名发布
2. 当 用户查看树洞帖子 时，系统应 显示内容但不透露作者身份
3. 当 用户评论树洞帖子 时，系统应 为每个帖子线程分配唯一的匿名标识符
4. 当 展示树洞内容 时，系统应 提供支持性互动选项（拥抱、支持图标）
5. 当 发生严重法律问题需要司法取证 时，系统应 保留加密的真实用户关联记录供最高权限管理员配合调查
6. 当 帖子作者在自己的树洞帖子下评论 时，系统应 将其匿名标识显示为"楼主"以便读者识别

### 需求 5: 求助板块

**用户故事:** 作为 用户，我想要 发布求助信息并获得帮助，以便于 我可以解决学习或生活中的问题。

#### 验收标准

1. 当 用户创建求助请求 时，系统应 要求选择分类标签（学习、生活、其他）
2. 当 用户查看求助请求 时，系统应 允许按分类和状态（开放/已解决）筛选
3. 当 帮助者提供解决方案 时，系统应 允许求助者将请求标记为已解决
4. 当 请求被标记为已解决 时，系统应 高亮显示被采纳的答案

### 需求 6: 市集板块

**用户故事:** 作为 用户，我想要 发布和浏览二手物品信息，以便于 我可以买卖闲置物品。

#### 验收标准

1. 当 用户创建市集商品 时，系统应 要求填写价格、描述和至少一张图片
2. 当 用户浏览市集 时，系统应 允许按分类和价格区间筛选
3. 当 用户联系卖家 时，系统应 提供私信沟通渠道
4. 当 商品售出 时，系统应 允许卖家标记为已完成并从活跃列表中移除
5. 当 用户进入市集交易页面 时，系统应 显示防诈骗安全提示
6. 当 卖家被多次举报欺诈 时，系统应 将其加入市集黑名单并禁止发布商品
7. 当 用户举报被核实为欺诈 时，系统应 扣除被举报者 20 信用分
8. 当 用户完成一笔成功交易 时，系统应 为买卖双方各增加 5 信用分
9. 当 用户信用分低于 60 分 时，系统应 限制其在市集板块发布商品
10. 当 用户每日活跃（登录/发帖/评论） 时，系统应 恢复 1 信用分（上限 100）

### 需求 7: 失物招领板块

**用户故事:** 作为 用户，我想要 发布失物或招领信息，以便于 我可以找回丢失物品或帮助他人。

#### 验收标准

1. 当 用户创建失物帖子 时，系统应 要求填写物品描述、丢失地点和丢失时间
2. 当 用户创建招领帖子 时，系统应 要求填写物品描述、发现地点和联系方式
3. 当 用户浏览失物招领 时，系统应 允许按类型（失物/招领）和时间范围筛选
4. 当 物品被认领 时，系统应 允许发布者将帖子标记为已解决

### 需求 8: 内容管理

**用户故事:** 作为 用户，我想要 管理我发布的内容，以便于 我可以编辑或删除自己的帖子。

#### 验收标准

1. 当 用户编辑自己的帖子 时，系统应 更新内容并记录编辑时间戳
2. 当 用户删除自己的帖子 时，系统应 从公开视图中移除该帖子
3. 当 用户查看个人主页 时，系统应 显示该用户创建的所有帖子
4. 当 用户收藏帖子 时，系统应 保存引用以便后续访问

### 需求 9: 数据持久化

**用户故事:** 作为 系统管理员，我想要 确保数据安全存储，以便于 用户数据不会丢失。

#### 验收标准

1. 当 系统存储用户数据 时，系统应 将数据持久化到 PostgreSQL 数据库
2. 当 系统缓存频繁访问的数据 时，系统应 使用 Redis 进行缓存
3. 当 系统序列化数据进行存储 时，系统应 使用 JSON 格式
4. 当 系统反序列化存储的数据 时，系统应 重建等效对象（往返一致性）

### 需求 10: 前端界面

**用户故事:** 作为 用户，我想要 使用美观易用的界面，以便于 我可以获得良好的使用体验。

#### 验收标准

1. 当 系统渲染 UI 组件 时，系统应 使用 DaisyUI 组件库
2. 当 用户在移动设备上访问系统 时，系统应 提供响应式布局
3. 当 用户切换主题 时，系统应 立即应用所选的 DaisyUI 主题
4. 当 用户执行操作 时，系统应 在200毫秒内提供视觉反馈

### 需求 11: 角色与权限分配

**用户故事:** 作为 系统管理员，我想要 给不同用户分配角色（如版主、管理员），以便于 授权特定人员协助维护社区秩序。

#### 验收标准

1. 当 系统管理员进入用户管理后台 时，系统应 展示所有用户及其当前角色列表
2. 当 管理员将"版主"角色分配给某用户 时，系统应 立即通过 Sa-Token 更新该用户的权限状态
3. 当 普通用户尝试访问受保护的管理接口 时，系统应 拒绝访问并返回 403 权限错误
4. 当 管理员封禁某个违规用户 时，系统应 强制该用户下线，并禁止其再次登录

### 需求 12: 内容审核与举报

**用户故事:** 作为 版主或管理员，我想要 在后台查看举报并处理违规帖子，以便于 保持社区环境的健康与合规。

#### 验收标准

1. 当 用户点击"举报"按钮并提交理由 时，系统应 标记该帖子并将其加入"待审核队列"
2. 当 版主查看审核队列 时，系统应 提供"保留"、"删除帖子"或"封禁作者"的操作选项
3. 当 帖子被版主删除 时，系统应 向原作者发送一条系统通知，说明删除原因
4. 当 新帖子包含系统预设的敏感词 时，系统应 自动拦截发布并转入人工审核状态

### 需求 13: 消息通知中心

**用户故事:** 作为 普通用户，我想要 及时收到关于我帖子的互动通知，以便于 我能快速回复评论或查看求助结果。

#### 验收标准

1. 当 其他用户点赞或评论了我的帖子 时，系统应 为我生成一条未读通知记录
2. 当 用户查看通知中心 时，系统应 高亮显示未读消息，并按时间倒序排列
3. 当 用户点击某条通知 时，系统应 标记该通知为"已读"并自动跳转到对应的帖子详情页
4. 当 管理员发布全站公告 时，系统应 将该消息广播给所有注册用户
5. 当 有新通知产生 时，系统应 通过 WebSocket/SSE 实时推送到客户端，无需用户刷新页面
6. 当 用户 WebSocket 连接断开后重连 时，系统应 自动拉取断开期间的未读通知（推拉结合策略）

### 需求 14: 全局搜索功能

**用户故事:** 作为 普通用户，我想要 通过关键词搜索全站内容，以便于 我可以快速找到感兴趣的话题、旧帖子或特定失物信息。

#### 验收标准

1. 当 用户在搜索栏输入关键词 时，系统应 检索所有公开板块（表白墙、求助、市集、失物招领）的标题和内容
2. 当 展示搜索结果 时，系统应 高亮显示匹配的关键词
3. 当 搜索结果包含"树洞"板块的内容 时，系统应 确保依然隐藏作者身份，不破坏匿名性

### 需求 15: 多媒体资源管理

**用户故事:** 作为 普通用户，我想要 上传图片来丰富帖子内容，以便于 更直观地展示失物细节或二手商品状态。

#### 验收标准

1. 当 用户上传图片 时，系统应 校验文件格式（仅限 JPG, PNG, WEBP）和文件大小（最大 5MB）
2. 当 图片上传成功 时，系统应 对图片进行压缩处理，以优化移动端的加载速度
3. 当 用户删除包含图片的帖子 时，系统应 标记关联的图片文件，并在后台进行异步清理（垃圾回收）以释放存储空间
4. 当 系统存储图片文件 时，系统应 使用 S3 协议兼容的对象存储服务（如 MinIO/阿里云 OSS），禁止使用应用服务器本地文件系统
5. 当 图片上传成功 时，系统应 异步调用内容安全 API 进行鉴黄/鉴暴检测
6. 当 图片被检测为违规内容 时，系统应 自动软删除该图片并封禁上传者账号
7. 当 图片超过 90 天无访问 时，系统应 自动转入低频存储（Cold Storage）以节省成本

## 非功能性需求（工程化与运维治理）

### 需求 16: 可观测性 (Observability)

**用户故事:** 作为 运维人员，我想要 通过链路追踪快速定位线上问题，以便于 在用户反馈问题时能够精准排查故障点。

#### 验收标准

1. 当 任意 HTTP 请求进入系统 时，系统应 为该请求生成唯一的 Trace ID 并贯穿整个调用链路
2. 当 请求经过 Controller/Service/Mapper 各层 时，系统应 在日志中自动携带 Trace ID（通过 MDC 实现）
3. 当 请求处理完成 时，系统应 记录请求耗时、响应状态码、请求参数摘要（脱敏后）
4. 当 发生异常 时，系统应 在日志中记录完整的异常堆栈并关联 Trace ID
5. 当 响应返回给客户端 时，系统应 在响应头中携带 X-Trace-Id 字段，便于用户反馈时提供追踪依据

### 需求 17: 高并发防护 (Resilience)

**用户故事:** 作为 系统管理员，我想要 系统具备限流和缓存防护能力，以便于 在流量高峰或恶意攻击时保持服务稳定。

#### 验收标准

1. 当 单个 IP 在 1 分钟内请求同一接口超过 60 次 时，系统应 返回 429 状态码并拒绝后续请求
2. 当 单个用户在 1 分钟内发帖超过 5 次 时，系统应 触发发帖限流并提示用户稍后再试
3. 当 热点数据（如热门帖子列表）被频繁访问 时，系统应 优先从 Redis 缓存读取，缓存未命中时才查询数据库
4. 当 缓存 Key 大量同时过期（缓存雪崩风险） 时，系统应 采用随机过期时间策略（基础 TTL + 随机偏移）
5. 当 查询不存在的数据（缓存穿透风险） 时，系统应 缓存空值或使用布隆过滤器拦截无效请求

### 需求 18: 代码质量管控 (Quality Gate)

**用户故事:** 作为 开发团队负责人，我想要 强制统一代码风格和质量标准，以便于 多人协作时保持代码一致性和可维护性。

#### 验收标准

1. 当 开发者提交代码前 时，系统应 通过 Git Hooks 自动执行代码格式化检查
2. 当 前端代码不符合 ESLint/Prettier 规则 时，系统应 阻止提交并提示具体错误位置
3. 当 后端代码不符合 Checkstyle 规则 时，系统应 阻止提交并提示具体错误位置
4. 当 代码被推送到远程仓库 时，系统应 触发 CI 流水线执行完整的代码检查和测试

### 需求 19: 持续集成/部署 (CI/CD)

**用户故事:** 作为 DevOps 工程师，我想要 实现自动化的构建、测试和部署流程，以便于 代码变更能够快速、安全地发布到生产环境。

#### 验收标准

1. 当 代码推送到 main/develop 分支 时，系统应 自动触发 CI 流水线
2. 当 CI 流水线执行 时，系统应 依次执行：代码检查 → 单元测试 → 属性测试 → 构建镜像
3. 当 所有测试通过 时，系统应 自动构建 Docker 镜像并推送到镜像仓库
4. 当 镜像构建成功且目标分支为 main 时，系统应 自动部署到生产环境（或等待人工审批后部署）
5. 当 CI/CD 任意阶段失败 时，系统应 发送通知到指定渠道（邮件/钉钉/企业微信）并阻止后续流程

### 需求 20: 性能与容量

**用户故事:** 作为 系统架构师，我想要 明确系统的性能指标，以便于 进行容量规划和压力测试。

#### 验收标准

1. 当 系统正常运行 时，系统应 支持 5000 日活用户 (DAU) 的并发访问
2. 当 流量达到峰值 时，系统应 能够承受 200 QPS 的请求压力
3. 当 用户执行常规操作（列表查询、发帖） 时，系统应 在 500ms 内返回响应
4. 当 执行全局搜索 时，系统应 在 1 秒内返回结果（数据量 10 万条以内）

### 需求 21: 数据安全与备份

**用户故事:** 作为 运维人员，我想要 确保数据安全和可恢复性，以便于 在发生故障时能够快速恢复服务。

#### 验收标准

1. 当 系统运行期间 时，系统应 每日自动执行数据库全量备份
2. 当 备份完成 时，系统应 将备份文件存储到独立的存储位置（非应用服务器）
3. 当 需要数据恢复 时，系统应 能够在 1 小时内恢复到最近一次备份状态
4. 当 备份文件超过保留期限（30天） 时，系统应 自动清理过期备份

### 需求 22: 数据生命周期管理

**用户故事:** 作为 系统管理员，我想要 自动管理冷数据，以便于 控制存储成本并保持系统性能。

#### 验收标准

1. 当 市集帖子超过 30 天无互动 时，系统应 自动将其标记为下架状态
2. 当 失物招领帖子超过 60 天未解决 时，系统应 自动归档并从活跃列表移除
3. 当 图片文件超过 90 天无访问 时，系统应 自动转入低频存储类型
4. 当 用户主动删除帖子 时，系统应 在 7 天后彻底清理关联的图片文件
